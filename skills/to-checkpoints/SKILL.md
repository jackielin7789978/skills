---
name: to-checkpoints
description: >
  在開始實作前根據對話內容或 plan 拆分檢查點，並在每個檢查點結束時停下來請使用者檢查並確認是否繼續。
---

根據目前的對話內容或 plan，拆分出大小適中的檢查點（checkpoints），在使用者確認後才開始實作。在每個檢查點結束時停下來，提示使用者檢查並確認是否繼續。確保實作過程中的每一步都符合使用者的期望。拆分完成產出一份可以跨 agent sessions 使用的文件。

**重要**：在每個檢查點結束時，請針對修改範圍做 linting 和 type checking。請簡單扼要總結完成的項目，並提供使用者檢查的範圍和方式，請勿擅自 commit。

**拆分要點**：每個檢查點必須能用 UI 手動測試，考慮到這是一個純前端專案，檢查點的拆分應該以功能或頁面為單位，確保每個檢查點都能獨立驗證其功能是否正常，UI 還沒實作的部分放 placeholder。錯誤範例：只寫好 API 呼叫函式但缺乏 UI 介面。

**禁止**：不要拆出「純新增檔案、不接線、UI 完全不變」的 foundation/preparation CP。使用者沒東西可以手動驗證，這種 CP 對 review 流程沒幫助。應該把基礎檔案合進第一個會實際接線或產生 UI 變化的 CP，讓每個 CP 都有可驗證的產出。例外：純新檔量極大、合進去會讓單一 CP 過肥時可說明理由，但預設不拆。
